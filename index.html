TOPIC 1: DATA-FLOW DIAGRAMS (DFDs)
1.1 What is a Data-Flow Diagram?

A Data-Flow Diagram (DFD) models the flow of information through a system and the activities (processes) that transform that information. It is one of the most commonly used tools in system analysis.

🗣️ In Plain English
Imagine water flowing through pipes in a building. The pipes are the data-flows (arrows), the taps and pumps are processes (what transforms the water), the tanks are data stores (where water is held), and the outside world — the street and river — are the external entities. A DFD maps that plumbing system for information instead of water.
✅ Key Benefit
DFDs are used because they are accessible to both technical staff AND non-technical users/customers. Both groups can look at the same diagram and understand the system.
1.2 The Four Symbols of a DFD
Symbols DFD — All Four Symbols Drawn Correctly
SymbolShapePurposeRules
ProcessRectangle with 3 sections (number | name | location)Transforms/uses data. The 'doing' part of the system.Must have at least 1 input AND 1 output. Named with action verb + object (e.g., 'Record Details').
Data-FlowNamed arrow (directed line)Represents data moving between elements.Always flows to/from a process. Labelled with a meaningful name. Dashed line = flow between external entities.
Data StoreLong rectangle with 2 sections (ID | Name)Where data is stored inside the system.Must have at least 1 input and 1 output flow. Use prefix: D=computer, M=manual, T=temporary.
External EntityOval/ellipse with identifierPeople or systems outside the system that provide or receive data.Always connected to a process by a data-flow. Duplicated entities get a diagonal cut corner.
🗣️ Real-Life Example
Think of a bank. The CUSTOMER (external entity) sends a DEPOSIT REQUEST (data-flow) to the PROCESS DEPOSIT process (process), which stores the record in ACCOUNT FILE (data store) and returns a RECEIPT (data-flow) to the CUSTOMER.
1.3 The Three Types of DFDs — The Hierarchy

DFDs are built in levels, from most general (top) to most detailed (bottom). This is called functional decomposition or levelling.

LevelAlso CalledWhat It ShowsKey Feature
Level 0Context DiagramThe ENTIRE system as ONE single process, plus all external entities and data-flows in/out.No data stores. No sub-processes. Shows system boundary only.
Level 1Level 1 DFDThe main subprocesses that make up the system. An 'exploded' view of the context diagram.Shows data stores. Processes numbered 1, 2, 3…
Level 2+Level 2/3 DFDBreaks down a single Level 1 process into its sub-processes.Processes numbered like 2.1, 2.2, 2.3… Must BALANCE with parent.
⚠️ Balancing Rule
Every data-flow that goes INTO a process in the parent diagram MUST appear in the child (lower level) diagram. No new inputs/outputs can be added without updating the parent too. This is called BALANCING.
1.4 DIAGRAM TYPE A: Context Diagram (Level 0)

The context diagram presents the entire system as a single process box, showing only what information comes in and goes out.

Diagram Type A Context Diagram (Level 0) — Hotel Booking System
Rules for Context Diagram Process numbered "0" NO data stores shown All external entities shown System = single process box
📋 Steps to Draw a Context Diagram
1. Identify external entities (who/what sends data in or receives data out).
2. Identify all data-flows in and out of the system.
3. Draw one big process box for the whole system.
4. Draw ovals for external entities.
5. Connect with labelled arrows.
1.5 DIAGRAM TYPE B: Level 1 DFD

The Level 1 DFD explodes the context diagram's single process into the main sub-processes of the system. Data stores now appear.

Diagram Type B Level 1 DFD — Estate Agency System (Course Case Study)
Note Process numbers (1, 2, 3…) do not imply sequence! All boundary flows must match the Context Diagram (Balancing Rule).
1.6 DIAGRAM TYPE C: Level 2 DFD (Functional Decomposition)

A Level 2 DFD zooms into one specific process from Level 1 and breaks it into its sub-steps. All data-flows into/out of the parent process must appear as boundary flows in the Level 2 diagram.

Diagram Type C Level 2 DFD — Decomposing "Loan of Video" (Process 2)
Key Rules for Level 2 Processes: 2.1, 2.2, 2.3 (derived from parent number) Duplicate entities/stores get diagonal/extra-line marks Asterisk (*) in bottom-right = primitive process (cannot decompose further)
📌 Decompose a process when:
(1) There are more than 6 data-flows around it, OR (2) The process name is too general/complex (e.g., 'Handle Customer Requests' — what does that really mean?). A process you cannot break down further gets an asterisk (*) in the bottom right corner.
1.7 Validation Checklist for DFDs

Before submitting/drawing a DFD in the exam, ask yourself:

  • Is every data-flow (arrow) attached to a process at one end?
  • Is every data-flow labelled with a meaningful noun?
  • Does every process have at least one input AND one output?
  • Is every process named with an action verb (e.g., 'Record Details', not 'Handle Stuff')?
  • Is every data store named in the plural (e.g., 'Customer Details', not 'Customer')?
  • Do duplicate entities/stores have a diagonal mark?
  • Do all levels BALANCE (same data-flows in/out as the parent)?
🗣️ Real-Life Scenario (Video Rental — the course example)
External Entities: Customer, Supplier. Processes: Create New Customer, Loan of Video, Stock Control. Data Stores: Customer File, Stock File. A customer (external entity) brings their membership card (data-flow) to 'Loan of Video' (process) which checks the Customer File (data store) and records Item on Loan in the Stock File (data store) before sending back a video loan (data-flow).
TOPIC 2: ENTITY-RELATIONSHIP DIAGRAMS (ERDs)
2.1 What is an ERD?

An Entity-Relationship Diagram (ERD) is a graphical tool for modeling the DATA of a system at a high level of abstraction. It shows what kinds of things (entities) exist in the system and how they relate to each other.

ERDs were invented by Peter Chen in 1976. The notation used in this course is from Elmasri and Navathe.

🗣️ In Plain English
Imagine you are building a school database. The THINGS you want to track — students, courses, teachers — are ENTITIES. The CONNECTIONS between them (a student TAKES a course, a teacher TEACHES a course) are RELATIONSHIPS. The details about each thing (student ID, name, age) are ATTRIBUTES. An ERD draws all of that out visually before you write a single line of database code.
2.2 The Core Building Blocks of an ERD
Symbols ERD — All Core Building Blocks
ElementSymbolMeaningExample
EntityRectangleA 'thing' or object with independent existence in the system. Represents a TYPE, not a single instance.employee, student, factory, book
AttributeOval connected to entityA property or characteristic of an entity.name, age, ID, salary
Key AttributeOval with UNDERLINED labelThe attribute that uniquely identifies each instance of the entity.Employee ID (no two employees share the same ID)
RelationshipDiamond connected to entitiesA meaningful connection between two or more entities. Like a 'verb' linking 'nouns'.employee WORKS AT factory
Weak EntityDouble-bordered rectangleAn entity that cannot be uniquely identified by itself — it needs its 'owner' entity.assembly_line needs factory to be identified (line 1 at Factory A vs line 1 at Factory B)
Identifying RelationshipDouble-bordered diamondThe relationship that a weak entity depends on for its identity.factory CONTAINS assembly_line
🔑 Key Distinction
An ENTITY is like a template or class (e.g., 'employee' represents ALL employees). An INSTANCE is one specific example (e.g., employee #47, John Doe). Attributes describe the template, not a specific person.
2.3 Cardinality Ratios — How Many?

Cardinality describes how many instances of one entity relate to how many instances of another. It is shown as numbers (1 or N) on the lines connecting a relationship to entities.

RatioMeaningReal-Life ExampleStatement
1:1One-to-OneEach country has exactly ONE capital city.Each country has 1 capital; each capital belongs to 1 country.
1:NOne-to-ManyOne teacher teaches MANY students.Each teacher teaches many students; each student has one class teacher.
N:MMany-to-ManyMany students take many courses.Each student takes many courses; each course is taken by many students.
Notation Cardinality & Participation Notation
2.4 Advanced ERD Concepts

2.4.1 Recursive Relationships

A relationship between an entity and ITSELF. Used when some instances of an entity are related to other instances of the same entity.

§2.4.1 Recursive Relationship — employee SUPERVISES employee

2.4.2 Composite Attributes

An attribute that is made up of smaller sub-attributes. For example, an address is composed of street, city, and postal code.

§2.4.2 Composite Attribute — address

2.4.3 Multi-valued Attributes

An attribute that can hold multiple values for one entity. Shown with a DOUBLE oval. Example: An employee can have multiple phone numbers. The 'phone' attribute would be a double oval, meaning one employee can have several phone numbers stored.

2.5 DIAGRAM TYPE D: Basic ERD — Factory System
Diagram Type D ERD — Employee, Factory & Assembly Line (Weak Entity)
Weak Key = number Full identity = factory city + line number (combined). A weak entity without its owner entity has no meaning.
🗣️ Real-Life ERD Example
For a university database: STUDENT (student_id, name, level) is related to COURSE (course_code, title, units) through an ENROLS IN relationship (N:M — many students enroll in many courses). LECTURER (staff_id, name) is related to COURSE through a TEACHES relationship (1:N — one lecturer teaches many courses). This is the ERD you would draw before building the university database system.
TOPIC 3: USE CASE DIAGRAMS
3.1 What is a Use Case Diagram?

A Use Case Diagram is a UML diagram that shows how users (actors) interact with a system to accomplish goals. It captures the FUNCTIONAL REQUIREMENTS of a system from the user's perspective.

🗣️ In Plain English
A Use Case Diagram is like a restaurant menu. The restaurant (system) offers dishes (use cases), and different types of customers (actors) can order different items. A regular customer can order food, while the chef (another actor) can update the menu. The menu is the system boundary.
FeatureDescription
Shows the WHATWhat can users do with the system? (not HOW the system does it internally)
User's perspectiveDescribes requirements from outside — what users need from the system
High-level overviewNot detailed — gives the big picture of all system functions
Functional requirementsEvery use case = one piece of functionality the system must provide
3.2 Use Case Diagram Notation
ElementSymbolMeaning
System BoundaryLarge rectangle enclosing the use casesDefines what is INSIDE the system vs. outside. Use cases go inside; actors go outside.
ActorStick figure with name belowA user, external system, or device that interacts with the system. Represents a ROLE, not a specific person.
Use CaseOval/ellipse with name insideA specific goal a user can achieve using the system. Represents a piece of functionality.
AssociationSolid line between actor and use caseShows that the actor participates in that use case.
IncludeDashed arrow labelled «include»One use case ALWAYS includes the behaviour of another. Like calling a subroutine that always runs.
ExtendDashed arrow labelled «extend»One use case SOMETIMES extends the behaviour of another. Optional extra behaviour.
GeneralizationSolid arrow with hollow triangle headInheritance. A child actor/use case inherits from a parent and can add more specific behaviour.
3.3 DIAGRAM TYPE E: Use Case Diagram — ATM System
Diagram Type E Use Case Diagram — ATM System
📌 Include vs. Extend — Easy Way to Remember
INCLUDE = mandatory helper. Like how 'Log In' is INCLUDED in almost every banking function — you always have to log in first.
EXTEND = optional add-on. Like 'Print Receipt' EXTENDS 'Withdraw Cash' — you can withdraw without a receipt, but the option exists.
3.4 Merits and Demerits of Use Case Diagrams
MeritsDemerits
Clear visual overview of all system functionsToo abstract — may oversimplify complex interactions
Easy for non-technical stakeholders to understandDoes not capture detailed business rules or logic
Helps identify system scope and boundariesOnly a static snapshot — doesn't show time or sequence
Guides requirements analysis from user's viewCannot show internal system processes or data flows
TOPIC 4: ACTIVITY DIAGRAMS
4.1 What is an Activity Diagram?

An Activity Diagram models the FLOW OF ACTIVITIES (steps/actions) within a system or process. It shows the sequence of actions, decision points, and parallel flows involved in completing a task.

🗣️ In Plain English
An activity diagram is like a detailed recipe or flowchart. For example, baking a cake: Start → Gather ingredients → Mix ingredients → [Decision: oven ready?] → Yes: Bake → Cool → Serve → End. You can also show that while the cake bakes, you can clean up (parallel activity — fork and join).
4.2 Activity Diagram Notation
ElementSymbolMeaning
Initial Node (Start)Solid filled circle ●The starting point of the process.
Activity/ActionRounded rectangleA task or step in the process (e.g., 'Enter Payment Details').
TransitionArrow between activitiesShows the flow of control — what happens next.
Decision Node (Branch)Diamond shape ◇A decision point where the flow splits based on a condition. Outgoing arrows labelled with conditions.
ForkThick horizontal bar (1 in / multiple out)Splits flow into PARALLEL paths. Multiple activities happen simultaneously.
JoinThick horizontal bar (multiple in / 1 out)Merges parallel paths back together. All parallel activities must complete before continuing.
Final Node (End)Solid circle inside a ring ◎ (bullseye)The ending point of the process.
4.3 DIAGRAM TYPE F: Activity Diagram — Online Shopping
Diagram Type F Activity Diagram — Online Shopping Process
🔑 Key Distinction from Use Case Diagrams
Use Case Diagrams show WHAT users can do (high level). Activity Diagrams show HOW a process flows step-by-step (detailed workflow). Use Cases are about GOALS; Activity Diagrams are about STEPS.
TOPIC 5: UML CLASS DIAGRAMS
5.1 What is a Class Diagram?

A Class Diagram is a STATIC structural diagram that shows the classes in a system, their attributes, their operations (methods), and the relationships between classes. It is the most important diagram in object-oriented design.

🗣️ In Plain English
A class is like a blueprint or cookie cutter. The CLASS 'Dog' defines what ALL dogs have in common: a name (attribute), a breed (attribute), and the ability to bark() (operation/method). An OBJECT is a specific dog — like your dog 'Bingo'. The class diagram shows all the blueprints and how they connect.

Created by combining the work of three researchers: Grady Booch (BOOCH), James Rumbaugh (OMT), and Ivar Jacobsen (OOSE) in the mid-1990s.

TOPIC 6: STATE DIAGRAMS
6.1 What is a State Diagram?

A State Diagram (also called a State Machine Diagram) models the DYNAMIC BEHAVIOR of a single object by showing all the possible states it can be in and how it transitions between those states in response to events.

🗣️ In Plain English
Think about your mobile phone. It can be: OFF, LOCKED, ACTIVE, ON A CALL, CHARGING. Each of these is a STATE. When you press the power button, you transition from OFF to LOCKED. When you unlock it, you go from LOCKED to ACTIVE. Events (pressing buttons, receiving calls) cause transitions between states. A state diagram draws all of this out.

All objects of the same class follow the same state diagram — it models their COMMON behavior.

6.2 State Diagram Notation
ElementSymbolMeaning
StateRounded rectangle with state nameA condition or situation where an object satisfies some invariant and waits for an event.
Initial StateSolid filled circle ●The starting state — where the object begins its life.
Final StateBullseye (solid circle inside ring) ◎The terminating state — the object's life ends here.
TransitionArrow between statesA change from one state to another triggered by an event.
Event LabelText on the transition arrowThe event that triggers the transition (e.g., 'card inserted', 'PIN entered').
Guard Condition[condition] in square bracketsA boolean condition that must be TRUE for the transition to occur.
Action/Effect/ action after event labelAn action that happens WHEN the transition fires (e.g., event [guard] / action).
6.3 DIAGRAM TYPE H: State Diagram — ATM Machine
Diagram Type H State Diagram — ATM Machine
📌 When to Use State Diagrams
State diagrams are most useful for objects whose behavior DEPENDS ON THEIR STATE and whose state CHANGES over time. Great for: ATM machines, traffic lights, order processing systems, vending machines, phone call handling.
TOPIC 7: SEQUENCE DIAGRAMS
7.1 What is a Sequence Diagram?

A Sequence Diagram shows how objects in a system INTERACT WITH EACH OTHER over TIME. It captures the sequence of messages exchanged between objects to carry out a specific use case or scenario.

🗣️ In Plain English
Imagine a phone call. You (Customer object) call the bank's automated system (ATM object). The bank asks for your PIN. You enter it. The bank verifies it with the database (Database object). The database says 'correct'. The bank shows your balance. You hang up. A sequence diagram maps this conversation from top to bottom in time order.
7.2 Sequence Diagram Notation
ElementSymbolMeaning
Object/ActorRectangle at top (ClassName:ObjectName)A participant in the sequence — an object, actor, or system.
LifelineDashed vertical line hanging from objectRepresents the object's existence over time (time flows downward).
Activation BarNarrow rectangle on lifelineShows the period when an object is active (executing a method).
MessageSolid arrow from sender to receiverA message/method call sent from one object to another.
Return MessageDashed arrow from receiver back to senderThe response/return value after a message is processed.
Self MessageArrow that loops back to the same lifelineAn object calling its own method.
Asynchronous MessageOpen arrowheadA message sent without waiting for a response.
7.3 DIAGRAM TYPE I: Sequence Diagram — ATM Transaction
Diagram Type I Sequence Diagram — ATM Transaction
📌 Sequence vs. State Diagram
State Diagram = ONE object across all its possible situations over its whole life.
Sequence Diagram = MULTIPLE objects interacting in ONE specific scenario from start to finish. Use sequence when asked about 'how objects interact in a use case'.

EXAM PREPARATION — LIKELY QUESTIONS & HOW TO APPROACH THEM
Section A: Theory Questions (Short Answer)
Q1 — Define a Data-Flow Diagram and state its four main elements.
Answer Guide: A DFD is a graphical representation of data flowing through a system. Four elements: (1) Process — transforms data, shown as rectangle; (2) Data-Flow — named arrow showing movement of data; (3) Data Store — rectangle where data is held; (4) External Entity — oval for things outside the system that send/receive data.
Q2 — Distinguish between a Context Diagram (Level 0) and a Level 1 DFD.
Answer Guide: Context Diagram = the whole system as ONE process, no data stores, shows only external entities and boundary flows. Level 1 = explodes the context into main sub-processes, adds data stores, shows detailed data flows. Both must BALANCE (same external data flows).
Q3 — What is meant by 'balancing' in levelled DFDs?
Answer Guide: Balancing means every data-flow entering/leaving a parent process must appear as a boundary flow in the child (lower-level) diagram that decomposes it. No new inputs/outputs can appear without updating the parent.
Q4 — State and explain the rules for drawing processes in a DFD.
Answer Guide: Rules: (1) Named with action verb + object (e.g., 'Record Customer'). (2) Must have at least one input AND one output. (3) Cannot act as source or sink alone. (4) Only processes that transform data are shown. (5) Process boxes have three sections: number, name, location.
Q5 — Define an Entity in an ERD. How is it different from an instance?
Answer Guide: An entity represents a TYPE of thing (e.g., 'employee' represents ALL employees). An instance is ONE specific example (e.g., employee #5, Amaka). The entity is drawn as a rectangle with the type name; instances are rows in the actual database table.
Q6 — Explain the three cardinality ratios in an ERD with examples.
Answer Guide: 1:1 — one country has one capital (one-to-one). 1:N — one department has many employees (one-to-many). N:M — many students take many courses (many-to-many). Shown as 1 or N on the line connecting relationship to entity.
Q7 — What is a weak entity? How is it represented in an ERD?
Answer Guide: A weak entity cannot be uniquely identified by its own attributes — it needs to be identified together with a related 'owner' entity. Represented by a DOUBLE-BORDERED rectangle. Its key is a partial/weak key (shown with DASHED underline). The identifying relationship is shown as a DOUBLE DIAMOND.
Q8 — Distinguish between «include» and «extend» relationships in Use Case Diagrams.
Answer Guide: «include»: One use case ALWAYS triggers another as a mandatory sub-behavior (e.g., 'Withdraw Cash' always «includes» 'Verify PIN'). «extend»: One use case OPTIONALLY extends another — it only happens under certain conditions (e.g., 'Print Receipt' «extends» 'Withdraw Cash' — not every user wants a receipt).
Q9 — What is a State Diagram? When is it most appropriate to use one?
Answer Guide: A state diagram models the dynamic behavior of ONE object across all possible states during its lifetime. Most appropriate for: objects whose behavior depends on their current state (ATM machines, traffic lights, phone systems, online orders). It shows states, transitions, events, and guards.
Q10 — Explain the purpose and components of a Sequence Diagram.
Answer Guide: Purpose: to show how objects in a system interact over time in a specific scenario. Components: Objects/Actors (top), Lifelines (dashed vertical lines), Activation bars (rectangles on lifelines), Messages (solid arrows), Return messages (dashed arrows). Time flows downward.
Section B: Diagram Drawing Questions
Q11 — Draw an ERD for a university: students, courses, lecturers.
Approach: Entities: STUDENT, COURSE, LECTURER. Relationships: STUDENT ENROLS IN COURSE (N:M). LECTURER TEACHES COURSE (1:N). Attributes: Student (studentID-key, name, level). Course (courseCode-key, title, units). Lecturer (staffID-key, name, department). Draw rectangles, diamonds, ovals with cardinality ratios on lines.
Q12 — Draw a Context Diagram for a hotel booking system.
Approach: Draw one large process box labelled 'Hotel Booking System' (number 0). Draw ovals for: GUEST, PAYMENT GATEWAY. Add labelled arrows: Guest → system (booking request, personal details), System → Guest (booking confirmation, receipt), System → Payment Gateway (payment request), Payment Gateway → System (payment confirmation).
Q13 — From the hotel booking context diagram, draw a Level 1 DFD with at least three processes and two data stores.
Approach: Processes: 1. Register Guest, 2. Process Booking, 3. Handle Payment. Data stores: D1 Guest File, D2 Booking File. Show all the same external entity flows from the context diagram. Add flows between processes and data stores. Number processes 1, 2, 3.
Q14 — Draw a Use Case Diagram for an Online Library Management System.
Approach: System boundary rectangle labelled 'Library System'. Actors: Student (outside left), Librarian (outside right). Use Cases inside: Search Book, Borrow Book, Return Book, Pay Fine, Register Member, Add Book, Remove Book. Show associations. 'Borrow Book' «includes» 'Verify Member'. 'Pay Fine' «extends» 'Return Book'.
Q15 — Draw a State Diagram for an online order in an e-commerce system.
Approach: States: Order Placed → Payment Pending → Payment Confirmed → Processing → Shipped → Delivered → [Final State]. Also show: Payment Failed (from Payment Pending), Cancelled (from Order Placed or Processing). Transitions labelled with events. Start with ● initial, end with ◎ bullseye.
Q16 — Draw a Class Diagram for a simple school management system with at least three classes and four relationships.
Approach: Classes: School, Student, Teacher, Course, Classroom. Relationships: School CONTAINS Classrooms (Composition — filled diamond). Student TAKES Course (Association, N:M). Teacher TEACHES Course (Association, 1:N). Student and Teacher generalize from Person (Generalization — hollow triangle). Show attributes and multiplicities.
Q17 — What are the four types of DFDs and explain functional decomposition?
Approach: Four models: (1) Current Physical DFD — how things work NOW physically. (2) Current Logical DFD — logical view of current system. (3) Required Logical DFD — what the new system SHOULD do. (4) Required Physical DFD — how the new system WILL be built. Functional decomposition = breaking a high-level process into sub-processes at lower levels, managed through the parent-child hierarchy of DFD levels.
Section C: Scenario-Based Questions
Q18 — Pharmacy dispensing system: identify external entities, data stores, processes, and draw the context diagram.
Approach: Read the scenario carefully. External entities = any person or system OUTSIDE the pharmacy (e.g., Doctor, Patient, Insurance Company, Drug Supplier). Data stores = where records are kept inside (Patient File, Prescription File, Drug Stock). Processes = actions done by the system (Receive Prescription, Dispense Drug, Manage Stock). Context diagram = one process box, all external entities as ovals, all boundary data-flows as arrows.
Q19 — Estate agent handles property listings and matches buyers. Draw a Level 1 DFD.
Approach: External entities: Client (property owner), Buyer. Processes: 1. Record New Client, 2. Record New Buyer, 3. Match Properties, 4. Invoice Client, 5. Archive Sale. Data stores: D1 Property File, D2 Buyer Details, D3 Invoices. Flows: property details from Client → Process 1 → D1. Buyer details from Buyer → Process 2 → D2. D1 + D2 → Process 3 → selected properties to Buyer. This is the Estate Agency case study from the course!
Q20 — Describe the difference between Aggregation and Composition with examples.
Answer Guide: AGGREGATION (open diamond): A whole-part relationship where the PART can exist independently of the whole. Example: A Department HAS Employees. If the department is deleted, the employees still exist. COMPOSITION (filled diamond): A strong whole-part relationship where the PART CANNOT exist without the whole. Example: A House HAS Rooms. If the house is demolished, the rooms cease to exist.
🎯 EXAM DAY STRATEGY
1. For diagram drawing questions — always label ALL elements. Use correct symbols.
2. For DFDs — check balancing between levels. Every external data-flow must appear on both the context and Level 1.
3. For ERDs — mark your key attributes with an underline. Use double borders for weak entities.
4. For all UML diagrams — remember what goes INSIDE vs. OUTSIDE boundaries.
5. For sequence diagrams — time flows downward. Draw lifelines as dashed vertical lines.
